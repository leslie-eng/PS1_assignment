Overview
This project is a PySpark ETL pipeline that ingests four raw, messy e-commerce CSV files, applies a series of data quality and transformation steps, and writes clean analytical outputs partitioned by year and month. It was built as part of the PSI Data Engineering Intern Take-Home Assessment.

---
Running the Pipeline
```bash
python pipeline.py
```
The pipeline runs all six tasks in sequence and writes outputs to `./output/`. It is fully idempotent — running it multiple times produces the same result because all writes use `mode('overwrite')`.
---
Pipeline Tasks
#	Task	Description
01	Ingestion & Schema Enforcement	Loads all four CSVs with explicit `StructType` schemas. Rows that fail casting are written to `output/rejected/` rather than silently dropped.
02	Data Quality & Cleaning	Removes duplicates, normalises mixed date formats (`YYYY-MM-DD` and `DD/MM/YYYY`) to ISO standard, standardises `customer_tier` to lowercase, drops rows with NULL `order_id` or `customer_id`, and flags negative `total_amount` values with a boolean column `is_negative_amount`.
03	Joins & Enrichment	Joins orders to customers (inner join) and to order items (left join). An anti-join isolates orphaned `order_items` which are written separately to `output/orphaned_items/`. Derives `net_amount = total_amount × (1 − discount_pct / 100)`.
04	Aggregations & Window Functions	Produces three window-based outputs: customers ranked by lifetime net spend within each country; a 7-day rolling order count per customer; and each product category's share of total revenue per calendar month.
05	Return Analysis	Joins returns to the enriched orders. Computes return rate per category and per `customer_tier`. Identifies the top 10 customers by total refund amount. Flags returns where `refund_amount > net_amount` with a boolean column `refund_exceeds_order`.
06	Output & Partitioning	Writes the final enriched dataset to Parquet partitioned by `order_year` and `order_month`. Writes all aggregation summary tables to CSV. All writes use `mode('overwrite')`.
---
Bonus Tasks Completed
Bonus	Description
B2 — Broadcast join	The customers table is small relative to orders and is broadcast in the orders→customers join. A comment in the code explains the reasoning.
B3 — DQ gate	Before the final write, assertions verify that no order has a NULL `customer_id` after joining, and that `net_amount >= 0`. The pipeline raises an exception and halts if either check fails.
B1 — Unit tests	Three pytest tests in `tests/test_pipeline.py` cover date normalisation, duplicate removal, and `net_amount` derivation using small synthetic DataFrames.
---
Data Quality Issues Handled
Issue	How it is handled
~8% duplicate rows	`dropDuplicates()` applied to all four tables after ingestion
Mixed date formats	`F.coalesce(F.to_date(..., 'yyyy-MM-dd'), F.to_date(..., 'dd/MM/yyyy'))` tried in sequence
~4% NULL `customer_id`	Rows dropped after deduplication; count logged before drop
~2% NULL `total_amount`	Rows dropped; count logged
~3% negative `total_amount`	Flagged with `is_negative_amount` boolean column — not dropped
~4% orphaned `order_items`	Isolated via anti-join and written to `output/orphaned_items/`
Inconsistent `customer_tier` casing	Normalised to lowercase with `F.lower()` and `F.trim()`
~5% refund anomalies	Flagged with `refund_exceeds_order` boolean column

Design Decisions
Native Spark functions over Python UDFs — all transformations use `pyspark.sql.functions` (`F.to_date`, `F.lower`, `F.trim`, `F.when`, `F.coalesce`, etc.). Python UDFs are avoided in the main pipeline because they break Spark's Catalyst optimiser and are significantly slower.
No `.collect()` inside the pipeline — `.collect()` and `.toPandas()` are only used inside pytest tests and for final console display. Calling them inside pipeline logic would pull all data to the driver and defeat the purpose of distributed processing.
Modular functions — each task is a separate function with a single responsibility. `pipeline.py` is the single entry point that calls them in order, making the flow easy to follow and individual steps easy to test.
Anti-join for orphaned items — a left anti-join on `order_id` between `order_items` and `orders` is the cleanest way to isolate rows that have no match. This is more readable and idiomatic than filtering on NULL keys after a left join.
7-day rolling window — implemented using `rangeBetween` with the date converted to a Unix timestamp in days, rather than `rowsBetween`, so the window is genuinely time-based regardless of how many orders fall within a day.
---
Assumptions
The pipeline is designed to run locally with a single-node SparkSession. No cluster or Databricks environment is assumed.
`discount_pct` is treated as a percentage value (e.g. `10` means 10%), so `net_amount = total_amount * (1 - discount_pct / 100)`.
Return dates are not validated against order dates. A return could technically precede its order in this dataset; this is noted as a known limitation.
Rows with a negative `total_amount` are kept in the enriched dataset but excluded from revenue aggregations in Task 04, since negative revenue would distort category share calculations.
---
Known Limitations
The 7-day rolling count uses `order_date` only, not order datetime. If multiple orders share the same date, they are all included in the window boundary.
The pipeline does not handle schema evolution — if new columns are added to the CSVs in future, the explicit `StructType` definitions in Task 01 will need to be updated.
Sample output files committed to this repo were generated with `--seed 42 --rows 2000`. Results will differ with different seeds.
